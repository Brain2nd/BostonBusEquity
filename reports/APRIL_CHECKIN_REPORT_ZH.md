# Boston Bus Equity 4月阶段汇报文档

## 本阶段改动摘要

这份文档重点总结 **4 月阶段新增的工作内容**。

相较于前期项目，本阶段最重要的改动是：

1. **新增实时延误推理链路**
2. **把项目模型接入 MBTA V3 官方实时 API**
3. **新增 baseline 延误评估用于量化对照**
4. **用真实历史数据和真实在线样例完成阶段验证**

后续各部分会围绕这些新增内容展开。

## 1. 项目概述与动机

### 1.1 项目主题

本项目围绕 **Boston MBTA 公交系统的服务公平性与延误预测** 展开。  
**4 月阶段新增的核心内容**是在前期离线分析基础上，进一步补充了两项关键能力：

1. **实时延误推理链路**
2. **baseline 延误对照测试**

本阶段目标不是只做更多离线图表，而是验证项目是否能够基于真实 MBTA 数据源支持一个合理的在线预测流程。

### 1.2 为什么这个阶段重要

前期项目已经关注了以下问题：

1. 不同线路、时段和服务场景下的延误模式
2. 不同线路和社区之间的服务质量差异
3. 公交表现与公平性问题之间的联系

4 月阶段进一步回答：

1. 当前仓库已有模型能否用于实时推理场景
2. 官方 MBTA 实时接口能否与项目模型可靠衔接
3. 是否能够补出一套简单但合理的 baseline，对模型结果形成解释性对照

因此，这个阶段非常适合作为 April check-in 的主体内容，因为它同时覆盖了数据、建模和系统实现。

## 3. 项目目标

### 3.1 目标 1：构建高质量实时推理功能

定义：

1. 接收一条实时或近实时公交记录
2. 从 MBTA 官方 API 获取所需输入
3. 用项目模型输出本地延误预测值

可测量标准：

1. 至少一条真实在线样例可以端到端跑通
2. 系统输出包含 `predicted_delay_minutes`
3. 输入来自真实 MBTA V3 API，而不是伪造样例

### 3.2 目标 2：补充 baseline 延误测试

定义：

1. 使用清晰的时间切分方式
2. 实现多个简单但合理的 baseline
3. 导出可比较的误差指标

可测量标准：

1. 成功生成 baseline 结果文件
2. 至少包含 5 个 baseline
3. 输出包含 RMSE、MAE、R2 等指标

### 3.3 目标 3：形成可用于 April check-in 的阶段叙述

定义：

1. 说明本阶段做了什么
2. 说明数据、模型和接口如何连接
3. 总结当前结果和下一步计划

可测量标准：

1. 有正式的中文和英文文档
2. 文档包含数据、方法和结果
3. 团队能够在 check-in 中解释关键设计决策

### 3.4 为什么这些目标是具体且可测量的

这些目标不是泛泛地说“分析数据”，而是明确到：

1. 使用哪个模型
2. 接入哪个实时 API
3. 使用什么时间切分
4. 输出哪些指标
5. 什么结果算是本阶段已经跑通

因此它们符合 rubric 中对 “specific and measurable” 的要求。

## 4. 数据采集计划

### 4.1 历史数据来源

历史数据仍然使用项目原有的 MBTA bus arrival/departure 数据，目录为：

```text
data/raw/arrival_departure
```

当前已确认：

1. 共 `61` 个 CSV 文件
2. 总体量约 `15.282 GB`
3. 年份覆盖 `2021-2026`

这说明 4 月阶段并没有切换到新的离线训练数据集，而是继续沿用项目原始核心数据。  
**本阶段的变化重点在实时输入链路，而不是更换历史训练数据。**

### 4.2 实时数据来源

实时输入使用 **MBTA V3 API**，主要接口为：

1. `schedules`
2. `predictions`

作用分别是：

1. `schedules`：获取当前之后的计划班次信息
2. `predictions`：获取 MBTA 当前实时预测结果

### 4.3 数据采集方式

历史数据采集方式：

1. 读取本地 MBTA 历史 CSV 档案
2. 进行离线预处理和建模准备

实时数据采集方式：

1. 基于 API 调用
2. 在推理时按请求访问 MBTA V3 API
3. 当前阶段不采用长期后台轮询任务

这也是本阶段的重要改动，因为前期工作主要是离线处理，而这一阶段明确加入了实时数据接入路径。

这满足了 Proposal Rubric 中对数据源和采集方式说明的要求。

## 5. 数据处理

### 5.1 本阶段的主要处理步骤

为了同时支持历史建模逻辑和实时推理，本阶段完成了以下数据处理：

1. 解析 `service_date`、`scheduled`、`actual`
2. 计算 `delay_minutes`
3. 过滤明显异常值
4. 规范化 `route_id`
5. 对齐历史数据与实时 API 的方向字段语义
6. 构造时间特征和统计特征

### 5.2 关键数据处理决策

#### 决策 1：实时推理选用 V2，而不是 V3

原因：

1. `V2` 特征可以在线构造
2. `V3` 的 wavelet、lag、rolling 特征更适合离线研究
3. 对于 stateless realtime inference，`V2` 更合理

#### 决策 2：继续使用仓库中已有模型

当前实时推理仍然使用仓库现有 checkpoint：

```text
models/delay_predictor_mlp_v2_lag_features_temporal.pt
```

这意味着：

1. 4 月阶段没有重新训练一个新的在线模型
2. 本阶段重点是把已有模型接入实时输入链路

这是 4 月阶段最需要重点说明的改动之一：

1. **没有更换模型本体**
2. **新增的是模型外围的实时推理能力**

#### 决策 3：构建 realtime bundle，而不是直接加载裸 checkpoint

realtime bundle 中统一打包了：

1. 模型权重
2. 模型配置
3. 特征顺序
4. scaler 参数
5. route、stop、direction 映射
6. 训练期统计特征

这样单条实时请求就能映射到与训练阶段一致的特征空间。

#### 决策 4：规范化 route_id

历史数据里 route_id 可能写成：

1. `01`
2. `1`

而实时 API 通常返回：

1. `1`

如果不统一格式，实时记录就无法正确匹配训练映射。因此本阶段显式做了 route ID 规范化。

#### 决策 5：对齐 direction 语义

历史数据使用：

1. `Inbound`
2. `Outbound`

实时 API 使用：

1. `0`
2. `1`

本阶段增加了显式映射，使实时 API 的方向值可以和历史训练数据保持一致解释。

### 5.3 真实数据子集策略

由于完整历史数据量较大，4 月阶段没有尝试直接对全量历史重新构建全部结果，而是采用 **真实数据子集策略**：

1. 仍然使用真实历史 CSV
2. 聚焦较新的 `2024`、`2025`、`2026`
3. 对每个文件抽样，用于阶段性 bundle 构建和验证

当前成功构建的 stage bundle 元数据为：

1. `years = [2024, 2025, 2026]`
2. `sample_per_file = 600`
3. `train_rows_used = 7200`
4. `test_rows_seen = 7800`

这样做的意义是：

1. 仍然基于真实历史数据
2. 阶段运行规模可控
3. 能形成足够可信的阶段汇报依据

## 7. 建模方法

### 7.1 项目整体尝试过的方法

在整个项目范围内，已经尝试过多条延误建模路线：

1. V1 baseline ANN
2. V2 lag-feature MLP
3. V3 wavelet-temporal 模型

而在 4 月实时阶段，实际用于在线推理的是：

1. `V2 MLP`

### 7.2 为什么 V2 MLP 适合本阶段

本阶段目标是 **高质量实时推理**，而不是只展示最高离线分数。

选择 `V2 MLP` 的原因：

1. 特征可以在线构造
2. 不需要服务端长期维护复杂滚动历史窗口
3. 在实时链路里更容易调试和解释
4. 对 stateless API 工作流更合适

### 7.3 本阶段增加的 baseline

**本阶段新补充的 baseline** 包括：

1. `zero_delay`
2. `global_mean`
3. `route_mean`
4. `stop_mean`
5. `hour_mean`
6. `route_hour_mean`

这些 baseline 的意义是：

1. 提供下界对照
2. 判断模型是否学到超过简单启发式的结构
3. 让模型结果更容易解释

## 9. 初步结果

这一部分汇总的是 **4 月阶段新增跑出的结果**。

### 9.1 Baseline 结果

本阶段 baseline 输出文件为：

```text
reports/delay_prediction_baselines_stage2_2024_2026.csv
```

结果如下：

| Baseline | RMSE | MAE | R2 |
|---|---:|---:|---:|
| `hour_mean` | 6.0679 | 4.4257 | -0.0258 |
| `global_mean` | 6.0769 | 4.4329 | -0.0288 |
| `route_mean` | 6.0985 | 4.4296 | -0.0361 |
| `stop_mean` | 6.1930 | 4.4530 | -0.0685 |
| `route_hour_mean` | 6.3473 | 4.5926 | -0.1224 |
| `zero_delay` | 6.8871 | 4.6646 | -0.3214 |

解释：

1. `hour_mean` 是当前阶段最强的 baseline
2. `zero_delay` 最弱
3. 更复杂的聚合 baseline 并不一定在当前子集上更好

### 9.2 实时推理结果

当前已经有至少一条真实在线样例端到端跑通。

输入：

1. `route_id = 1`
2. `stop_id = 64`
3. `direction_id = 1`

从 MBTA V3 获取到：

1. `scheduled_time = 2026-04-25T11:09:00-04:00`
2. `scheduled_headway = 10.0`
3. `MBTA live predicted delay = -6.2167 minutes`

项目本地模型输出：

1. `predicted_delay_minutes = -0.3711 minutes`

这个结果表明：

1. 官方实时 API 路径是通的
2. 实时输入能够转成模型特征
3. 本地模型能够产出实时预测
4. 项目模型输出可以与 MBTA 官方实时预测做对照

这并不代表当前模型已经优于 MBTA 官方实时预测，  
但它明确证明了 **本阶段新增的实时推理主链路** 已经在真实输入上打通。
