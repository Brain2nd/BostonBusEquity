# Boston Bus Equity 第二阶段进展报告（中文）

## 1. 本阶段目标

本阶段的重点不是重做完整离线研究，而是集中完成两项可用于第二次汇报的工作：

1. 开发高质量的实时推理功能
2. 补充 baseline 延迟测试，并给出可解释的结果

本报告只围绕这两个目标展开。

## 2. 本阶段是否已跑通真实数据

结论：**已经跑通真实数据，但目前采用的是“真实数据子集 + 官方实时接口”的阶段性方案。**

### 已确认的真实数据来源

本地已经存在项目原始 MBTA bus arrival/departure 历史数据：

1. 位置：`data/raw/arrival_departure`
2. 文件数：`61` 个 CSV
3. 总体积：约 `15.282 GB`
4. 覆盖年份：`2021-2026`

这些都是真实历史数据，不是伪造样例。

### 已确认的实时接口

本阶段实时输入使用的是 **MBTA V3 API**，重点使用：

1. `schedules`
2. `predictions`

其中：

1. `schedules` 用来获取当前之后的计划到站/发车时间
2. `predictions` 用来获取 MBTA 官方当前实时预测时间

这符合项目所需的“真实实时接口”方向，而不是只做一个假接口演示。

## 3. 本阶段完成的核心实现

## 3.1 实时推理链路

本阶段已经实现了项目内的本地实时推理能力，主要包括：

1. 从本地历史数据构建 realtime bundle
2. 加载现有 `V2` MLP checkpoint
3. 将单条实时请求转换为模型输入特征
4. 提供本地 API 接口
5. 支持直接请求预测
6. 支持先查询 MBTA V3 API，再把结果转为模型输入进行预测

### 为什么选 V2 而不是 V3

本阶段在线模型采用 `V2 MLP`，而不是 `V3`。

原因是：

1. `V2` 的特征是因果可在线构造的
2. `V3` 使用 wavelet、lag、rolling 等更偏离线研究的特征
3. 如果直接把 `V3` 当作 stateless 在线模型，会在方法上不严谨

所以从“汇报质量”和“方法正确性”的角度看，本阶段选择 `V2` 作为实时推理模型是合理的。

## 4. 本阶段真实运行的数据子集策略

由于原始历史 CSV 总量较大，直接全量重跑会显著拖慢阶段结果产出，因此本阶段采用了**真实数据子集**方案：

1. 使用真实历史 CSV，而不是模拟数据
2. 聚焦最近年份：`2024, 2025, 2026`
3. realtime bundle 构建时采用每文件采样

本次实际成功构建的 stage2 bundle 元数据如下：

1. `years = [2024, 2025, 2026]`
2. `sample_per_file = 600`
3. `train_rows_used = 7200`
4. `test_rows_seen = 7800`
5. `source_parquet_exists = False`

这意味着：

1. 本阶段结果来自真实数据
2. 但还不是完整 15GB 全量训练/评估结果
3. 它适合作为本月阶段汇报的工程验证结果

## 5. Baseline 结果

本阶段 baseline 结果文件：

```text
reports/delay_prediction_baselines_stage2_2024_2026.csv
```

结果如下：

| Baseline | RMSE | MAE | R2 |
|---|---:|---:|---:|
| `hour_mean` | 6.0679 | 4.4257 | -0.0258 |
| `global_mean` | 6.0769 | 4.4329 | -0.0288 |
| `route_mean` | 6.0985 | 4.4296 | -0.0361 |
| `stop_mean` | 6.1930 | 4.4530 | -0.0685 |
| `route_hour_mean` | 6.3473 | 4.5926 | -0.1224 |

### 结果解读

1. 在本阶段的真实数据子集上，`hour_mean` 是表现最好的 baseline
2. `zero_delay` 明显最差，说明“永远预测不延误”不是一个合理 baseline
3. `route_hour_mean` 没有优于简单 `hour_mean`，说明当前子样本下更细粒度聚合没有稳定带来提升

从汇报角度，这组 baseline 结果已经足够支撑“我们补上了对照组，并且拿到了真实数值结果”。

## 6. 实时推理结果

## 6.1 官方实时接口验证

我们已经验证：

1. `schedules` 接口可以返回当前之后的计划班次
2. `predictions` 接口可以返回 MBTA 当前实时预测记录
3. 两者可以联合用于构造在线推理输入

### 一个真实在线样例

使用：

1. `route_id = 1`
2. `stop_id = 64`
3. `direction_id = 1`

在本阶段运行时刻拿到的真实在线记录为：

1. `scheduled_time = 2026-04-25T11:09:00-04:00`
2. `scheduled_headway = 10.0`
3. `mbta_prediction_delay_minutes = -6.2167`

基于本地 stage2 realtime bundle，项目模型输出为：

1. `predicted_delay_minutes = -0.3711`

### 结果解释

这个结果说明了三件事：

1. 官方实时接口链路已经真实可用
2. 本地模型已经能够消费这条实时输入并输出预测值
3. 我们现在可以把“项目模型预测”与“MBTA 官方当前预测”并排比较

这正是第二次汇报里最有价值的工程结果之一。

## 7. 本阶段发现并修复的关键问题

为了保证这条实时链路不是“看起来能跑”，本阶段已经定位并修掉几类关键数据对齐问题：

### 7.1 route_id 前导零问题

历史数据中有些线路写成 `01`，而实时接口返回 `1`。

如果不规范化：

1. 历史数据里的 route mapping 会和实时接口对不上
2. 在线推理会出现假性 `Unknown route_id`

本阶段已经修复：

1. 历史数据读取时统一 route_id
2. runtime 中对实时输入也统一 route_id

### 7.2 direction_id 语义不一致问题

历史数据使用：

1. `Inbound`
2. `Outbound`

实时接口返回：

1. `0`
2. `1`

本阶段已显式加入方向映射逻辑，使实时接口输入能转成训练期语义。

### 7.3 schedule 查询逻辑问题

初版逻辑没有限定“当前之后”的计划记录，导致拿到的可能是当天更早的班次。

本阶段已修复：

1. `schedules` 查询加入 `filter[min_time]`
2. 只使用当前之后的真实计划记录
